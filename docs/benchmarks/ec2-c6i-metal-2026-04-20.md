## Baseline Concurrent Get Anomaly Investigation

The baseline concurrent Get throughput (18.1M ops/s, 0.22x ratio) is **not a fair comparison**. Root cause analysis:

**BaselineCache::Get()** (`BaselineCache.h:90`) calls `store.find(key)` without taking any lock — it is lock-free by accident under the warmup-then-read benchmark pattern. On real hardware, `unordered_map::find()` on a stable map is ~54ns per call.

Verification test on the same c6i.metal instance:

| Method | Throughput (4 threads) |
|--------|----------------------:|
| `unordered_map::find()` with no lock | **18.6M ops/s** |
| `unordered_map::find()` + `shared_lock` | 9.0M ops/s |

The 18.6M matches the baseline's 18.1M exactly, confirming the baseline benefits from unsynchronized `unordered_map::find()`.

**UB concern:** This is the same data race documented in §5.9 — `unordered_map::find()` during concurrent `insert()` is undefined behavior. The benchmark warmup-then-read pattern avoids it, but production mixed read/write workloads would trigger UB. Phase 2a's CMap (seqlock + CAS) was designed specifically to resolve this limitation.

**Fair comparison:** The concurrent **Set ratio of 1.89x** is the fair metric — both Furrballs and baseline take locks on the write path, and Furrballs wins through lock partitioning.

## Run 2: Thread Scaling (same instance, same session)

64-byte values, thread-local routing, local Gets. Each thread count uses a fresh FurrBall (overwrite=true). 10 iterations averaged. Thread pinning distributes threads round-robin across 2 NUMA nodes via `PinCurrentThreadToNode(t % numNodes)`.

| Threads | Set ops/s | Get ops/s | Threads/Node | Set Scaling (vs 4t) | Get Scaling (vs 4t) |
|---------|----------:|----------:|------------:|--------------------:|--------------------:|
| 4       | 3,875,618 | 5,891,794 | 2           | 1.00x               | 1.00x               |
| 8       | 3,334,534 | 7,221,196 | 4           | 0.86x               | 1.23x               |
| 16      | 2,365,749 | 9,033,473 | 8           | 0.61x               | 1.53x               |
| 32      | 1,323,892 | 9,227,251 | 16          | 0.34x               | 1.57x               |
| 64      |   810,975 | 10,127,801 | 32          | 0.21x               | 1.72x               |

**Key observations:**
- **Get throughput scales positively (+72% from 4→64 threads):** CMap's seqlock-based Find is nearly contention-free, validating the per-node lock partitioning thesis for reads.
- **Set throughput degrades (-79% from 4→64 threads):** All writes on a node contend on the same SpinLock. At 32 threads/node, SpinLock contention dominates. The documented upgrade path is per-thread PromoteBuf sharding (§2.8.5 in whitepaper).
- **Get scaling plateaus at ~16 threads (8/node):** 9.0M→9.2M→10.1M beyond 16 threads shows diminishing returns, likely from CMap's shared ctrl array probe traffic and memory bandwidth saturation.
